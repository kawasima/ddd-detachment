## 7.1　三つの性質

唯識思想には、あらゆる存在を三つの性質（三性、trisvabhāva）から捉える枠組みがあります。遍計所執性（へんげしょしゅうしょう）、依他起性（えたきしょう）、円成実性（えんじょうじっしょう）の三つです。

第2章でも触れましたが、これは三種類の「もの」があるという分類ではありません。同じ一つのものを、三つの異なる眼で見る枠組みです。

この章では、三性説をソフトウェアのモデリングに適用します。そこで浮かび上がるのは、モデリングにおける根本的な転倒——**ノード（実体）と関係のどちらが先か**——という問題です。

## 7.2　依他起性 — 関係こそがモデルの実質である

三性の中で最初に論じるべきは依他起性です。依他起性は三性の**基盤**だからです。遍計所執性は依他起性の上に生じ、円成実性は依他起性から遍計所執性を除いたものです。

### すべては関係の網の中にある

依他起性とは、すべてのものは他の条件に依存して（依他）生起する（起）という性質です。何ものも単独では存在しません。条件（縁）の網の中にあります。

> **本書での読み替えについての注意。** インド唯識における依他起性は、本来、識の転変が因縁によって生起することを指す概念で、識のあり方に踏み込んだ規定です。一般的な「関係存在論」（あらゆる存在を関係の網として捉える哲学的立場）と、そのまま同じものではありません。本書では、依他起性を**ソフトウェア設計の場面に持ち込む際には「関係性」として読み替える**、という方針を採ります。以降の節で「関係の網」と語るとき、それは設計上の読み替えとしての関係であり、唯識の依他起性の全体ではないことを念頭に置いてください。

ソフトウェアシステムは、この依他起性の構造を色濃く持っています。一つのHTTPリクエストが処理される過程を考えてみましょう。リクエストはロードバランサを経由し、Webサーバに到達し、ミドルウェアを通過し、ルーターによって振り分けられ、コントローラがドメインロジックを呼び出し、リポジトリがデータベースにクエリを発行し、結果がシリアライズされてレスポンスになります。どの部品も、他の部品との関係の中でのみ意味を持ちます。

### モデリングにおける関係の優位

ここでモデリングについて考えます。「顧客」とは何でしょうか。

多くのモデリングでは、まず「顧客」というエンティティが存在し、そこに「注文」「住所」「支払い方法」との関係が付属する、と考えます。顧客が先、関係が後です。

しかし依他起性の観点から見ると、順序が逆です。「注文する」「届け先がある」「支払う」——これらの関係の網目の交差点に、「顧客」というノードが浮かび上がるのです。関係が先であり、ノードは関係の結節点に過ぎません。

第1章で引いたWilliam Kentの洞察を思い出しましょう。エンティティの境界は対象に内在せず、観る側の視点に依存していました。これを依他起性の言葉で言い直せば、**エンティティとは、依存関係の網目から切り出された見かけ上の結節点**だということになります。

この「関係が先、ノードが後」という見方は、モデリングの実践に直接的な含意を持ちます。エンティティを先に定義してからリレーションシップを足すのではなく、業務における関係の網目を先に観察し、その中から必要な結節点を切り出す。この順序の転換が、依他起性に基づくモデリングの態度です。

これは「エンティティから書き始めるな」という実装手順の話ではありません。実装の便宜上はテーブル定義やクラス定義から手を付けることもありますし、それ自体は問題ありません。ここで言っているのは、モデルが行き詰まったとき、あるいは設計を疑うときに、**ノードの定義そのものを「関係の網目から切り出されたもの」として問い直す視点を持つ**ということです。手順としてどこから書き始めるかと、何が一次的か（ノードか、関係か）の見方は、別のレイヤーの話だと考えてください。

### 複雑さは移動する

依他起性の観点から、設計における重要な観察が得られます。

システムの本質的な複雑さは消えにくく、境界の切り方を変えると別の場所に形を変えて現れることが多いものです。あるモジュールを分割すれば、モジュール内の複雑さは減りますが、モジュール間の関係——インターフェース、プロトコル、データ変換——が増えます。複雑さが消えたように見えても、別の場所に移っているだけ、ということが起きます。

ここで注意すべき区別があります。よい抽象化や境界設定によって、**偶発的複雑性**（accidental complexity）——本来不要なのに実装の都合で生じている複雑さ——は実際に減らすことができます。一方、対象となる業務そのものに由来する**本質的複雑性**（essential complexity）は、境界を変えても消えるわけではなく、どこかに残ります。本書が「総量が保存される」と言いたいのではなく、「本質的な複雑さは、境界を変えても別の場所に現れやすい」と言いたいのは、この区別を意識してのことです。

依他起性が言っているのは「すべてが関係の網の中にある」ということです。すべてが関係の中にあるという見方を持っていれば、分割によって関係が消えるのではなく形を変えて現れる、ということを予期しやすくなります。設計とは複雑さを完全に消すことではなく、減らせる偶発的複雑性は減らしつつ、残る本質的複雑性を適切な場所に配置することだ——という態度は、依他起性の見方と親和的です。依他起性から演繹的に導かれるわけではありませんが、関係の網を先に見るレンズを持つことで、この態度に落ち着きやすくなる、という関係です。

## 7.3　遍計所執性 — ノードを実体視する罠

### 関係からノードへの転倒

依他起性の観点では、関係の網が先であり、ノードはそこから浮かび上がる結節点でした。しかしわれわれは、ノードの方を先に見て、関係を付属物のように扱いがちです。

遍計所執性とは、名前や概念をものごとに投射し、その投射を実在するものとして執着してしまう性質です。モデリングにおいては、**ノード（エンティティ）を独立した実体として固定化し、関係をその付属物として扱う態度**が遍計所執性にあたります。

この転倒は至る所で起きます。

### 飛べない鳥問題 — 分類の実体視

オブジェクト指向設計で有名な「飛べない鳥問題」は、この転倒の典型例です。

`Bird` クラスに `fly()` メソッドを定義する。鳥は飛ぶものだ——と、われわれの認知が告げます。しかしペンギンは鳥であり、飛べません。

問題は「鳥」という分類を実体視したことにあります。「鳥」というノードが先にあり、そこに「飛ぶ」という性質が本質的に結びついている——と見なした。しかし依他起性の観点では、「飛ぶ」「羽がある」「卵を産む」「くちばしがある」といった性質の関係の束があるだけであり、「鳥」はそれらの性質の交差点に名前を付けたものです。性質の束の中のどれが必須でどれが偶有的かは、「鳥」というノードに内在しているのではなく、文脈に依存しています。

解決は、振る舞いベースの抽象（`Flyable` というcapability）と分類ベースの抽象（`Bird` というtaxonomy）を分離することです。これは、ノードから出発するのをやめ、**関係（振る舞い）の側からモデルを組み立てる**ことを意味しています。

### エンティティ思考の罠

もう少し日常的な例を挙げましょう。ECサイトの設計で「注文」をモデリングするとき、「注文エンティティ」をまず定義し、そこに属性（注文日、合計金額、ステータス）と関係（顧客、注文明細、配送先）を足していく——これは自然なアプローチに見えます。

しかし、この「注文エンティティ」は遍計所執性を含んでいます。「注文」はそれ自体として独立に存在するわけではありません。「注文」は、顧客が商品を選び、支払い方法を指定し、配送先を伝えるという**関係の結節点**として浮かび上がるものです。

「注文」を独立したエンティティとして実体視すると、問題が生じます。販売チームにとっての「注文」と物流チームにとっての「注文」は、異なる関係の網目から浮かび上がる異なる結節点です。にもかかわらず「注文は注文だ」とノードを実体視すると、一つのエンティティに異なる文脈の関係を詰め込むことになり、モデルが肥大化します。

これは第4章で論じたBounded Contextの問題でもあります。文脈ごとにモデルを分けるとは、「ノードは文脈によって異なる結節点である」ことを認めることです。

### コードで見るノード先行と関係先行

具体的なコードで両者の違いを見てみましょう。

```java
// ノード先行 — 「注文」という実体を先に定義する
class Order {
    OrderId id;
    Customer customer;
    List<OrderLine> lines;
    ShippingAddress shippingAddress;
    PaymentMethod payment;
    OrderStatus status;
    ShipmentTracking tracking;
    // 販売・物流・決済の関心が一つのノードに集まる
}
```

「注文」を独立した実体として先に置くと、あらゆる文脈の関係がこのノードに集約されます。販売の関心も、物流の関心も、決済の関心も、すべてが一つの `Order` に詰め込まれます。

```java
// 関係先行 — 業務上の関係から結節点を切り出す

// 「顧客が商品を購入する」関係 → 販売文脈の結節点
record SalesOrder(OrderId id, CustomerId customer,
                  List<OrderLine> lines, Money total);

// 「注文品を届け先に届ける」関係 → 物流文脈の結節点
record Shipment(OrderId orderId, Address destination,
                List<PackageItem> items);

// 「注文の代金を受け取る」関係 → 決済文脈の結節点
record Payment(OrderId orderId, Money amount,
               PaymentMethod method);
```

関係から出発すると、「注文」は文脈ごとに異なる結節点として現れます。販売にとっての「注文」と物流にとっての「注文」は、異なる関係の網目から浮かび上がる異なるものです。一つの万能な `Order` エンティティは必要ありません。

ここで、三つの `record` がいずれも `OrderId` を共有していることに気づいた読者がいるかもしれません。「結局どこかで共通の `Order` を前提しているのではないか」という疑問です。ここで共有されている `OrderId` は、**文脈間の契約が剥き出しになった形**だと理解してください。万能 `Order` に畳み込まれていたときには見えなかった契約が、関係先行の分割によって表面化している、ということです。場合によっては文脈ごとに独立した識別子（`SalesOrderId`・`ShipmentId`・`PaymentId`）を持たせ、それらの対応を別途イベントや相関キーで表現する方が、文脈の独立性をより正確に表現できます。ここでの `OrderId` は暫定的な共有識別子であり、その適切な形は次節で扱う「文脈間の関係の設計」の一部として決まります。

依他起性の観点に立てば、これは自然なことです。関係の網が先にあり、ノードはそこから切り出されるのですから、切り出す文脈が異なれば結節点の姿も異なります。

### 分割は始まりに過ぎない

ここで一つ注意しておきます。万能な `Order` を `SalesOrder` / `Shipment` / `Payment` に分けただけでは、関係先行のモデリングが完成したことにはなりません。むしろ、本当に設計の重みがかかってくるのはこの分割の後です。

万能 `Order` がうまく隠していたのは、**文脈をまたぐ関係そのもの**でした。同じ一つのオブジェクトに販売・物流・決済を詰め込めば、文脈間の整合性は同じインメモリの構造の中で（少なくとも見かけ上は）一意に決まります。これを分割すると、文脈間の関係が剥き出しになり、設計上の問いが姿を現します。

- **文脈間の契約。** `SalesOrder` の `OrderId` を `Shipment` や `Payment` がどう参照するか。販売文脈の `OrderLine` と物流文脈の `PackageItem` の対応はどう決めるか。これは公開されたインターフェース、イベントスキーマ、共有カーネルといった形で明示する必要があります。
- **イベントの方向と意味。** 「注文された」「支払われた」「出荷された」「配送完了」「キャンセルされた」——どの文脈がどのイベントを発し、どの文脈がそれを受けて反応するか。イベントの命名と方向は、関係の網のどの辺をどちら向きに通すかの決定です。
- **整合性境界。** 一回のトランザクションでまとめて変えるのは、どこまでか。販売と決済を同じトランザクションに収めるのか、決済は非同期に整合させるのか。集約境界はトランザクションの境界とほぼ重なります。
- **失敗時の補償。** 「支払いが成功した後に在庫切れが判明したら何が真実か」「キャンセルは販売・決済・物流のどこまで遡って取り消されるか」「部分的に成功した処理をどう巻き戻すか」——ハッピーパスでは消えていた問いが、分割すると必ず浮上します。

これらは関係先行の設計に**おまけ**として付いてくる作業ではありません。「結節点の切り出し」と「文脈間の関係の設計」は、関係先行モデリングの両輪です。むしろ後者の方が運用品質に直結します。`record` を分けたところで設計を終わりにせず、文脈間の関係そのものを設計対象として扱うこと——これが、依他起性に基づくモデリングが実装に降りたときの実際の姿です。

## 7.4　円成実性 — 実体視の空を見る

円成実性とは、遍計所執性を離れて依他起性をありのままに見たときに現れる、ものの真実の姿です。別の何かが新たに存在するのではありません。依他起性から実体視（遍計所執性）を取り除いたものが円成実性です。

ここで誤読を一つ防いでおきます。「真実の姿」「ありのまま」と聞くと、設計上の唯一の正解、ベストアーキテクチャ、絶対的に正しいモデルが存在することを示唆しているように響くかもしれません。**そうではありません**。円成実性は何かポジティブな正解を提示する概念ではなく、「実体視を外したときに残る見え方」を指す否定的な定義の概念です。設計に置き換えるなら、「これが正しいモデルだ」を見つけるための概念ではなく、「自分のモデルが実体視に陥っていないかを点検する」方向に向く概念だと理解してください。

### 円成実性の構造

ここでもう少し正確に述べておきます。世親の『唯識三十頌』は円成実性を「依他起性の上に、遍計所執性が常に存在しないこと」（於彼依他起　常遠離前性）と定義しています。また『成唯識論』では「二空所顕の真如」——人空（自己に固定的な実体がないこと）と法空（ものごとに固定的な実体がないこと）によって顕わになる、もののありのままの姿——と説明されます。

注意すべきは、円成実性が「常」（nitya）と形容されている点です。依他起性は条件によって生じ滅する無常なものですが、円成実性——依他起性の上に遍計所執性が存在しないということ——は、われわれが気づこうと気づくまいと、常にそうです。関係の網をノードに固定化するのはわれわれの側の遍計所執性であり、関係の網そのものにはもともとノードの実体性は存在しません。この「もともと存在しない」という事態は、時間や条件に左右されず常に成り立っています。

円成実性は高度な修行の果てに体得される境地であり、設計原則のレベルにそのまま落とし込むことはできません。しかし、円成実性の構造——実体視が取り除かれたとき、依他起性がそのまま見える——には、設計に対する独自の示唆があります。

### 実体視を取り除くと何が見えるか

設計の文脈で考えてみましょう。

「完璧なモデル」を追い求めること自体が遍計所執性です。「正しいエンティティの定義」がどこかに存在し、それを見つけ出せばモデリングは完成する——この態度は、結節点の一つの切り出し方に実体性を付与しています。不確かな将来に備えて過剰な抽象化を作り込むことも、まだ現れていない関係の網目を実体視することにあたります。根拠のある予測に基づく設計まで否定しているわけではありません。問題は、不確かな将来を確実であるかのように見なして設計に反映することです。

これらの実体視を取り除いたとき、何が残るのでしょうか。

長期間にわたって運用・改修されてきたシステムの中に、不思議と安定している部分があることがあります。過度な抽象化が削ぎ落とされ、かといってべた書きでもなく、業務の関係の構造がそのままコードの構造に映し出されている。ノードの定義に力みがなく、関係の網目に沿って自然に切り出されている。こうした部分は、度重なる要件変更を経てもなお壊れにくい傾向があります。

これは円成実性の体現ではありません。しかし、実体視を取り除いた設計が持つ性質を示唆しています。実体視がないということは、特定の切り出し方への固執がないということです。固執がなければ、文脈の変化に応じて結節点を組み替えることへの抵抗が小さくなります。

円成実性が「常」と形容されるのは、もののありのままの姿がいつでもそこにあるからです。関係の網はいつでもそのまま存在しており、見えなくしているのはわれわれの遍計所執性の側です。設計が改善されるとき、何かが新たに「加わる」のではなく、余計な実体視が「取り除かれる」。この方向性は、円成実性の構造と通じています。

## 7.5　三性説による設計の問い

三性説は、設計を吟味する際の実践的な問いを提供します。これは本書の提案であり、唯識の原典に設計レビューの方法論があるわけではありません。しかし三性の構造は、モデリングの問い直しに有効なレンズになると考えます。

**遍計所執性の問い：** この設計は、ノードを実体視していないか？

- 「このエンティティは独立した実体か、それとも関係の結節点か？」
- 「このモデルの境界は、業務の関係の網目に沿っているか、それとも設計者が持ち込んだ区切りか？」
- 「このエンティティに複数の文脈の関心が詰め込まれていないか？」

**依他起性の問い：** この設計は、関係の網を正しく認識しているか？

- 「このモジュールが依存しているものは、すべて見えているか？」
- 「この分割により、複雑さはどこに移動したか？」
- 「隠れた結合——明示されていない依存関係——はないか？」

**円成実性の問い：** この設計は、ノードの実体視を離れてものごとをそのまま見ているか？

- 「取り除いても問題ないエンティティや抽象化はないか？」
- 「この設計で、今解くべき問題は解けるか？」
- 「この設計は、設計者の不安ではなく、実際の関係の構造に応えているか？」

## 7.6　三性は分離しない

最後に強調しておきたいのは、三性は三つの独立した世界ではないということです。

ものごとは関係の網の中にあります（依他起性）。われわれはその網の中から結節点を切り出し、名前を付けて実体化します。その実体化が関係の網の実態に即しているか、過度な固定化に陥っていないかを問い続けること（遍計所執性の点検）。そして固定化を手放して関係の網をそのまま見ること（円成実性の方向）。

この三つは、モデリングの一つの活動の中に常に同時にあります。依他起性（関係の網）なしにはモデリングの対象がなく、遍計所執性（結節点の固定化）なしにはコードに書き表せず、円成実性（固定化の相対化）なしには文脈の変化に対応できません。

そして、第5章で論じたように、自分のモデリングが特定の立場から行われていることは末那識の働きによって見えにくくなっています。三性の問いを立てる自分自身もまた、ある立場から問いを立てています。この自覚を持つことが、三性説を設計に活かす出発点です。
