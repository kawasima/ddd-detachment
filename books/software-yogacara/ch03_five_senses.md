## 3.1　前五識とは

第2章で見た通り、前五識は外界と直接接触する五つの識です。ここではその性質をもう少し詳しく見ていきましょう。

前五識が生じるには、三つの条件が揃う必要があります。**根**（こん）——感覚器官、**境**（きょう）——対象、**識**（しき）——認識の働き、の三つです。眼識は、眼根（眼という器官）と色境（色彩や形という対象）を縁として生じます。器官だけがあっても、対象がなければ識は生じません。対象だけがあっても、器官がなければ同様です。

| 識 | 根（器官） | 境（対象） |
|----|-----------|-----------|
| 眼識 | 眼根 | 色（しき）— 色彩や形 |
| 耳識 | 耳根 | 声（しょう）— 音 |
| 鼻識 | 鼻根 | 香（こう）— 匂い |
| 舌識 | 舌根 | 味（み）— 味わい |
| 身識 | 身根 | 触（そく）— 触れた感覚 |

前五識には二つの重要な特徴があります。

第一に、前五識は対象を**直接的に了別します**。この直接的な了別を「現量」（げんりょう）と呼びます。眼識は赤い色をそのまま知覚しますが、それを「リンゴだ」と概念的に判断することはしません。概念を介した判断は第六意識の仕事です。

第二に、前五識が生じるには根と境が揃うだけでなく、**作意**（さくい）——注意を向けるという心の働き——も必要です。目の前にあっても注意を向けていなければ見えません。しかしいったん条件が揃えば、前五識は対象を選り好みしません。眼識は美しいものも醜いものも、同じように知覚します。

## 3.2　ソフトウェアの前五識

### 外界からの入力

前五識のソフトウェアにおける対応物は、**システムが外界から入力を受け取る接触面**です。

**APIエンドポイント。** HTTPリクエストが到着し、バイト列としてのペイロードがシステムに届きます。この段階で、システムの「眼」にあたるものが生のデータを受け取ります。

**メッセージキュー。** 他のシステムやサービスからイベントメッセージが届きます。Kafkaのコンシューマー、SQSのポーリング——これらはシステムの外界との接触面です。

**ファイル入力。** バッチ処理で取り込むCSVファイル、アップロードされたドキュメント。システムにとっては外界から届いた生のデータです。

**物理センサー。** IoTデバイスの温度センサー、GPS、カメラ。文字通り物理的な「根」を通じて「境」を知覚しています。

ユーザーインターフェース（UI）はやや特殊な位置づけです。UIはシステムと人間の**接触面**であり、人間の前五識（眼識・身識など）とシステムの入力受付が出会う場所です。システムの側から見れば、UIから届くのもHTTPリクエストやWebSocketメッセージなどの生のデータです。

### 生のデータ — 前五識が受け取るもの

前五識がもたらすのは、概念化される前の直接的な知覚です。ソフトウェアにおいてこれは、ひとまず**生のバイト列**に対応すると考えてください。実装上は UI、API ゲートウェイ、ミドルウェア、シリアライザなど複数の層が間に挟まりますし、バイト列はすでに TCP やフレーミングといった解釈を経て届いています。「生のバイト列」は厳密な一対一対応ではなく、**業務概念に変換される前の接触面**を考えるための比喩です。

```
POST /api/orders HTTP/1.1
Content-Type: application/json

{"customer_id": "C-1234", "items": [{"product_id": "P-5678", "qty": 2}]}
```

このHTTPリクエストは、システムにとっての「感覚データ」です。厳密に言えば、ここに到達するまでに TCP のフレーミング、HTTP のパース、ヘッダの解釈などすでに多層の処理が走っており、メソッドやパス、ヘッダといった HTTP の構造はすでに読み取られています。この時点で「業務概念としてはまだ何も判断されていない」と言い換えてください。`/api/orders` というパスが「注文の作成」という業務上の意図に対応するか、ボディが「正当な注文」を表しているか——こうした業務的な意味づけは、この後の処理で初めて行われます。

前五識が赤い色を現量として知覚するがそれを「リンゴ」とは判断しないように、APIが受け取ったデータは、それが業務的に何を意味するかをまだ「知らない」状態です。

## 3.3　境界の位相 — 生のデータから概念へ

### 前五識と第六意識の区別

ソフトウェアにおける興味深い問題は、前五識的な受信（生のデータ）と第六意識的な分別（概念的なモデリング）の境界をどこに置くか、という点です。

唯識では、前五識と第六意識（五倶意識）はほぼ同時に働きます。眼識が赤い色を知覚するのとほぼ同時に、第六意識が「リンゴだ」と判断します。両者は時系列に並ぶ段階ではなく、同時に生起する別々の働きであり、分析上区別されるものです。

ソフトウェアでも、この区別は実行時には一瞬のうちに渾然一体となってしまいがちですが、設計の観点からは明示する意義があります。この区別に対応するプラクティスが「Parse, Don't Validate」です。

```java
// 接触面で受け取る生のデータ
String rawJson = request.getBody();

// パース — ドメインの概念へ引き渡す
OrderRequest order = OrderRequest.parse(rawJson);
```

`String` から `OrderRequest` への変換は、単なる型の付け替えではありません。接触面に届いたデータを、ドメインモデルが扱える分別の形に引き渡す境界の通過です。型はこの境界を明示する手段として機能しています。

この変換は一方向のものとして扱うべきです。`OrderRequest` に変換したデータを「元の生文字列として再解釈する」ことは、型システムが保証していた構造的な前提——「このデータは OrderRequest として妥当である」——を無効化します。設計原則として「パース済みデータを生データとして扱い直してはいけない」という直観が働くのは、このためです。ただし、生データ自体をログやイベントストアに保存することは別の話で、これは接触面で受け取ったものをそのまま蔵しておく行為にあたります。

### バリデーションの位相

ここで、バリデーションの位相についても整理しておきましょう。一口に「バリデーション」と言っても、参照している前提が層ごとに違います。Parse, Don't Validate の発想に沿って、入力データの検証を四つの層に分けて見ていきます。

**(1) 表現フォーマットの解釈** — そもそもデータが想定の形式として読めるか。HTTP リクエストとして読めるか、ボディが JSON として読めるか、などです。多くの場合フレームワークやライブラリが担います。「読めない」段階での失敗は、業務概念にすら届く前の失敗です。

**(2) スキーマ・型の検証** — 必須フィールドが揃っているか、各フィールドの型が合っているか、列挙値が想定の範囲か。OpenAPI、JSON Schema、静的型などで表現される層です。先ほどの「パース」がここまでをカバーします。

**(3) 値オブジェクトの局所的不変条件** — `Quantity` が正の整数であること、`Money` が通貨と金額の整合した組であること、`EmailAddress` が形式的に妥当であること——**ある一つの値が、その値だけで成立すべき条件**を満たしているか。これは外部状態を参照しない、値オブジェクトのコンストラクタが保証すべき不変条件です。Parse, Don't Validate の核心は、この (3) までを「型に持たせる」ことで、それ以降のコードからチェックを排除することにあります。

**(4) ドメイン状態に依存する検証** — 顧客が実在するか、在庫があるか、現在の注文ステータスから「キャンセル」へ遷移できるか——**現在のドメイン状態を参照しないと判定できない**条件です。これは値オブジェクト単体では完結せず、リポジトリやドメインサービスを介した判定になります。

この四層を、前五識と第六意識の区別におおまかに寄せて読むとすれば、次のようになります。(1) は前五識的な「届いた」段階、(2)(3) は外部入力の文脈で働く第六意識の分別（受信と並行するという意味で五倶意識に近い側面を持ちます）、(4) は現在のドメイン状態を能動的に参照する第六意識の判断です。ただし厳密には、(3) の値オブジェクトの不変条件は入力の有無と切り離して型の内部論理としても成立しうるもので、五倶意識と独頭意識の境界にまたがる処理です。ここでの対応は時系列的な段階の一致ではなく、「接触面に近い分別」と「ドメイン状態を参照する分別」を分ける視点として受け取ってください。

この四層を混ぜると、関心の分離が崩れます。(3) を「ドメイン層の意味的バリデーション」として後段に押し出すと、不変条件を持たない値が業務ロジックの中を流れることになり、随所でチェックが要ります。一方で (4) を「入力検証」と呼んで境界に押し付けると、ドメイン状態への依存が境界層に染み出します。境界では (1)(2)(3) を通して値を**確実に妥当な型に変える**こと、ドメイン層は (4) のように現在の状態を参照する判定だけを担うこと——この分担が、前五識と第六意識の分析的な区別に対応しています。

## 3.4　前五識の性質とシステム境界

### 選り好みしない受容

先に述べたように、前五識は条件が揃えば対象を選り好みせずに知覚します。美しいものも醜いものも、同じように知覚する。知覚した内容に対して「これは良い」「これは悪い」と判断するのは、第六意識以降の仕事です。

APIエンドポイントにも同じ性質があります。リクエストが到着すれば、正当なリクエストであれ攻撃であれ、システムはまずそれを受け取ります。その内容が妥当かどうかの判断は、受け取った後の処理（バリデーション、認証、認可）が担います。

この性質は「入力は信用できない」という設計原則につながります。システムの外界との接触面では、あらゆる入力が到着しうることを前提に設計する必要があります。SQLインジェクション、XSS、パストラバーサル——これらはすべて、外界から届いたデータが内部のプロセスで意図しない形で解釈されることで起きる問題です。

ただし「まず受け取る」と書くと、現実のシステム像とはずれます。実際の入力経路には、WAF、TLS終端、APIゲートウェイ、認証ミドルウェア、スキーマ検証など複数の境界があり、不正な入力はそれぞれの段で拒否・正規化されていきます。前五識との対比で「受け取った後の処理が判断する」と言うときも、それは単一の関門ではなく、層をなす処理の総体だと考えてください。

対策も、ひとまとまりの「サニタイゼーション（入力の無害化）」として括ってしまうと、現代の実装感覚からは雑になります。実際には、用途に応じて異なる処理が必要です。境界では、入力を厳密に**パース**して構造を確定し、文字コードや表記ゆれを**正規化**します。そのうえで**認証・認可**によって誰の何の操作かを判定します。さらに、危険は出口にも生じます。SQLは**パラメータ化クエリ**で組み立て、HTMLやJavaScript、シェル、URLなど**出力先の文脈ごとに適切なエスケープ／エンコード**を行うことで、データがコードとして解釈されることを防ぎます。「入力の段階でまとめて無害化する」という発想だけでは、文脈別エンコードの重要性が見えなくなります。

これらの処理は、規則に基づく変換と判断を伴う以上、前五識そのものの働きではなく、第六意識の活動に属します。ただし、その目的は業務的な分別（「この注文は有効か」）ではなく、生のデータを安全に扱えるようにすること、および出力先で安全な形に整えることです。ソフトウェアの設計上は業務ドメインの分別を取り囲むように配置される処理ですが、これは唯識自体に第六意識の層分けがあるわけではなく、ソフトウェア側の整理の問題です。

### 多様な入力チャネル

人間が五つの感覚を通じて異なる種類の対象を知覚するように、ソフトウェアシステムも複数の入力チャネルを通じて異なる形式のデータを受け取ります。

```
Web UI (HTML Form) ──→ ┐
Mobile API (JSON)  ──→ │
Batch File (CSV)   ──→ ├→ ドメインモデル（第六意識）
Message Queue      ──→ │
gRPC               ──→ ┘
```

唯識において、前五識はそれぞれ異なる対象（色・声・香・味・触）を知覚しますが、それらはすべて第六意識（五倶意識）によって統合されます。同様に、ソフトウェアの複数の入力チャネルは異なる形式でデータを受け取りますが、ドメイン層は統一されたモデルで処理します。

ヘキサゴナルアーキテクチャ（ポートとアダプター）は、この構造を設計パターンとして表現したものです。各入力チャネルにはアダプタが置かれ、外界から届いた多様な形式のデータを、ドメインモデルが扱える形に変換します。このアダプタは前五識そのものではありません。前五識で受け取った生のデータを、ドメインモデル（第六意識による分別）が扱える形に整える変換の仕組みです。唯識の中に同じ位置付けの概念はなく、異質な形式のデータを統一的に扱う必要があるがゆえにソフトウェア側で自然に現れる層だと言えます。

## 3.5　前五識の限界と意義

前五識は概念を形成しません。判断もしません。記憶もしません。しかし、**外界についての認識**は、前五識を経由せずには始まりません。外界を知る窓は前五識だけであり、他の識が外界を扱うには必ずこの窓を経なければなりません（独頭意識による思考・想像・夢は前五識なしに働き、末那識と阿頼耶識は絶えず活動し続けています。前五識は外界への窓であって、心の働きすべての起点ではありません）。

ソフトウェアにおいても、外界とやりとりするシステムは、入力を受け取る接触面から始まります。API、メッセージキュー、ファイル入力——これらの接触面の設計は「浅い」仕事に見えるかもしれません。しかし、ここでの設計判断は、外界と接する処理のすべてに影響します。

どのような形式でデータを受け取るか。パースの段階で何を保証するか。生データをどこまで保存するか。これらは前五識的な層の設計判断ですが、第六意識以降のモデリングの自由度を規定します。

## 3.6　設計レビューで使う問い

本章の内容を、設計レビューで実際に使える問いに落とし込んでおきます。

**接触面の問い：** この境界では、業務概念に変換される前の素材を扱えているか？

- 「接触面で受け取るデータと、ドメインの概念は分けて表現されているか？」
- 「業務的な意味づけが、入力境界に紛れ込んでいないか？」
- 「生データ（リクエスト原文、イベント原文）を、必要に応じて遡れる形で保存しているか？」

**Parse, Don't Validate の問い：** 妥当性が、後段から見て型として保証されているか？

- 「(1) フォーマット解釈、(2) スキーマ・型、(3) 値オブジェクトの局所的不変条件、(4) ドメイン状態に依存する検証——どこで何を担っているか説明できるか？」
- 「値オブジェクトのコンストラクタが (3) を保証しているか？それともドメイン層に漏れているか？」
- 「(4) を入力境界に押し付けて、ドメイン状態への依存が境界に染み出していないか？」

**選り好みしない受容の問い：** 接触面は、想定外の入力にも壊れずに対応できるか？

- 「不正・悪意ある入力に対して、どの層が拒否や正規化を担っているか？」
- 「入力の段階で『無害化』に頼り、出力時の文脈別エンコードが弱くなっていないか？」

次の第4章では、前五識が受け取った生のデータを、第六意識がどのように分別し、概念化し、モデルとして構造化するかを論じます。
