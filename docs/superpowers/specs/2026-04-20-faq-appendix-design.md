# 設計書: 巻末付録「FAQ: 従来のDDD解説との違い」

- 日付: 2026-04-20
- 対象書籍: `books/ddd-detachment/`

## 1. 目的

従来ネット記事で広く解説されている「DDD + クリーンアーキテクチャ」の典型パターン (Controller/Service/Repository の3層、@Entity をドメインモデルとして使う、Bean Validation でドメインモデルを守る、DTOで詰め替える、Aggregate/Repository/UseCase の語彙など) しか学んでいない読者が、本書の設計を読んだときに抱く**用語・概念のズレに由来する困惑**を、Q&A形式で解消する。

本書は「古典ドメインモデルパターンの解脱」として以下を主張している:
- ドメインモデルは `record` + `sealed interface` で十分
- バリデーションは Raoh デコーダが境界で一度だけ行う (Always-Valid Layer)
- レイヤー間の詰め替えは最小化し、ドメインモデルをそのまま受け渡す
- 振る舞いは別クラス (SubscriptionBehavior) に分離する

従来の典型と上記の主張はいくつもの点で食い違う。巻末付録FAQはその食い違いを明示し、読者が本書を腹落ちして読み終えられるよう支援する。

## 2. 対象読者

- 従来のQiita/Zenn系「DDD+クリーンアーキテクチャ」解説記事の典型だけを学んでいる読者
- 本書の主張と自分の常識が食い違うことに**困惑している**読者
- (対象外: 懐疑的な読者への反論、特定フレームワーク (Spring/JPA) の運用論)

## 3. 問い選定の方針

本 FAQ は現実の困惑にフォーカスする。具体的には、DDDコミュニティで実際に寄せられた質問 ([little-hands/ddd-q-and-a](https://github.com/little-hands/ddd-q-and-a/issues) の 798 件) を精査し、以下を条件に問いを選定した:

1. **Issue として複数回(3件以上)出現する頻出テーマである** (現実性)
2. **本書のアプローチが明確かつ独自な答えを返せる** (本書固有性)
3. **本書のスコープ (単一サービス内部のレイヤー設計) に収まる** (スコープ)

戦略的 DDD、モデリング方法論、組織論、特定 SaaS 運用論 (Spring Security・SendGrid 詳細など) は対象外。Issue に出現しない抽象論(「Always-Valid Layer と Value Object は何が違うのか」「DIPは成立しているのか」「ヘキサゴナル対応」など)も、現実の困惑ではないと判断して落とした。

### 7.1 各問の方向性メモ

執筆時のガイド。本文での触れ方を簡潔に示す。

**Q1. Form / Command / Entity を分けるのが「正しい詰め替え」では?**
典型解説ではレイヤー間DTO分離が「関心の分離」の具体形として示される。実装上は「カラム1つの追加で複数箇所を直す」「レイヤーごとにクラスを作ると詰め替えが面倒」という苦情が頻出する (#774, #540)。本書は、詰め替えの数ではなく**どのレイヤーがどの型の知識を持つべきか**で設計を評価する立場。9章の2-way / 1-way Mappingの区別で整理し、「DTOがゼロになる主張ではない」「レイヤー間 DTO としての Form/Command は不要だが、外部APIのレスポンス形式などのDTOは別目的で引き続き使う」を明示する。

**Q2. デコーダは結局バリデーションと何が違うのか**
現場の具体的困惑: (a) 画面フィールドを赤く光らせたいのでコンストラクタで例外を投げるとUIで使いにくい (#727)、(b) 複数フィールドのエラーを一度に返したい (#708)、(c) ドメイン例外を独自にすべきか IllegalArgumentException か (#713)、(d) マスタ存在チェックは VO 生成時か (#734)、(e) ID 存在確認はビジネスロジックではないか。本書の立場: バリデーションは「検査して真偽を返す」、デコーダは「パースして型を確定させる」。デコーダは **`ValidationFailure` を値として集約**するため、例外フロー設計が不要。ID存在確認も**形式的正当性の一部**としてデコーダに置ける(6章の形式的正当性/業務的正当性の表を参照)。

**Q3. ドメインEntity と ORM Entity は分けるべきか?**
典型の Spring Boot 構成では `@Entity` がドメインモデルを兼ねる。一方 DDD 書籍では「分けるべき」とも言われ、**実務で混線する**質問が非常に多い (#728, #597, #299, #55)。本書の立場: ドメインモデルは `record` + `sealed interface` で書き、ORM Entity クラスは**作らない**。Repository 実装が `ResultSet` や jOOQ レコードを直接 record に変換する。詰め替え専用の中間クラスは不要。2章で示した「外側から作る」設計の逆流を防ぐ核心。

**Q4. Optional / nullable な属性や「既存データで値がない属性」をどう表現するか**
「必須だが旧データに値がないので null 許容にするしかない」(#760, #736)、「未設定/未入力をどう書くか」(#359, #531) が頻出。本書の立場: `Optional<T>` や null ではなく、**sealed interface でライフサイクル段階を型として分離**する。例: `Subscription.Draft` / `Subscription.Active` / `Subscription.Suspended` のように、必須属性が揃っていない段階を別 record ヴァリアントで表現する。7章の状態遷移モデルの延長として位置づける。

**Q5. `record` のドメインモデルは「振る舞いを持つ」原則に反するのでは?**
「ドメインモデル=データ+振る舞いを一体にしたクラス」という直感への対応。DomainService も「ドメインオブジェクトのメソッドにすれば済むのでは」という声がある (#634)。本書は**振る舞いを別クラス (Behavior)** に分離することで、データの変更理由(列の追加・削除)と振る舞いの変更理由(業務ルールの変化)を独立させる。関心の分離の一形態であることを示す。

**Q6. sealed interface での状態遷移は State パターンや enum 分岐と何が違うのか**
現場では「ステータスを enum で持ち if-else 分岐するコード」が典型的に登場する。仮登録/本登録ユーザー (#47, #48)、タスクのライフサイクル (#693)、組み込み制御 (#98) など複数の形で質問される。State パターンは**現在の状態を切り替える仕組み**、sealed interface による状態表現は**状態ごとに型が異なる**もの。コンパイル時網羅性チェックの利点を、通常のState パターンや enum 分岐では得られない点を示す。

**Q7. 振る舞いクラスは Domain Service と何が違うのか**
ユースケースとドメインサービスの境が曖昧、という質問が頻出 (#745, #750, #634)。Domain Service は通常「複数エンティティをまたぐロジック置き場」として定義される。本書の Behavior クラスは**特定の record に対する状態遷移と業務ルール検査**を担当し、対応する record と1:1で紐づく。DomainService は複数集約に跨る特殊ケースに限定、ApplicationService はトランザクション境界と横断処理に限定、という責務表を示す。

**Q8. 権限・認可ロジックはドメイン層・UseCase層・Controllerのどこに書くか**
ddd-q-and-a で頻出のテーマ (#787, #220, #183, #252 系列)。update ユースケースに認可コードが重複する、AOP で切り出すべきか、という悩みが多い。本書の立場: 認可の**許可/不許可を sealed 戻り値として型で表す** (例: `Result<PostDeleted, PermissionDenied>`)。AOP/横断処理ではなく、Behavior が sealed 戻り値を返し、Controller がレスポンスに射影する。権限判定ロジックは Behavior に集約し、update ごとに if 文を書かない。

**Q9. 部分更新 (PATCH) のようにフィールド単位で更新したい場合、どう設計するか**
「Entity全体を取得→書き戻しがコスト高」「loginId と password で更新タイミングが違う」(#289, #315)、「update ユースケースに色々詰め込まれる」(#288, #506) が頻出。本書の立場: 属性セットごとに**小さな Behavior (ユースケース) を切る**。record の `with` 的置換で差分適用。入力は必要な部分だけのデコーダで受ける。Repository の保存単位は集約ルートだが、ユースケースの入力契約は細かくする。

**Q10. Aggregate (集約) はどこに行ったのか**
本書は「集約ルートを1つの可変クラスにする」という典型表現を採らない。**不変 record による状態の表現**で、集約が保つべき不変条件はデコーダと Behavior で担保する。集約の概念自体を否定するのではなく、表現方法が変わる。

**Q11. Aggregate Root・Repository・トランザクション境界の対応は?**
「Repository にトランザクション制御を公開するのは?」(#790)、「削除できる/できないをドメインで表現したい」(#794)、部分更新の保存単位 (#315) などが質問として存在。本書の立場: Repository は引き続き存在するが、**不変 record を保存/復元する**形に変わる。トランザクション境界は ApplicationService (Behavior を呼び出す層) に残る。集約の整合性はデコーダ+Behaviorが保証する。

**Q12. Value Object は `record` に吸収されたのか**
「コードを持つ属性はEntityかVOか」(#726)、「エンティティのフィールドは原則VOか」(#754)、「値が決まっているVOはEnumでよいか」(#704)、「VOの不変・交換可能の意味」(#784) など質問が散見。実質的には `record` が VOの性質 (不変・値ベース等価性) を最初から備える。従来「Value Object をクラスで書いていた」苦労が消える。値が決まっているケースでは `sealed interface` が Enum より適切なこともある。

**Q13. Factory パターンとデコーダの関係は?**
「新規作成と再構成でチェックが違う」(#77, #569)、「Factory の IF はドメイン、実装は Repository か」(#686)、「Entity は Factory, VO はコンストラクタ」という判断 (#681)。Factoryは「正しい状態のオブジェクトを生成する責任」を持つ。デコーダは**入力をパースしてドメイン型を確定させる**責任を持つ。後者は前者の一形態と見なせる。本書では「新規作成のデコーダ」と「DB再構成のデコーダ」を別定義することで、チェック粒度が違うケースに自然対応する。

**Q14. UseCase / Interactor 層はどこに相当するのか**
本書の ApplicationService (Behavior 呼び出し + トランザクション制御) が UseCase に相当する。UseCase を廃止しているわけではなく、**ドメインに振る舞いを持たせすぎない**ことで UseCase がシンプルになる。DomainService との境 (#745) にも1パラグラフで触れる (Q7 との連続性で)。

**Q15. Gateway・Presenter はどこに対応するのか**
Gateway = Repository 実装 / 外部 API Client (インフラ層)、Presenter = Controller 内でのレスポンス整形 or 専用 Presenter クラス。外部 API の置き場質問 (#779 SendGrid, #203 メール送信, #735 S3) が頻出。本書の立場: 外部境界は Gateway (出力アダプター) とし、**外部レスポンスもデコーダで record 化する**。Repository と Gateway は概念分離するが、どちらもドメイン層に interface、インフラ層に実装。Spring Security や SendGrid の運用詳細には踏み込まない。

**Q16. Controller でドメインモデルを直接扱うのは層の漏れでは?**
「Controller で TaskStatus Enum を参照してよいか」(#721)、「API仕様/画面仕様はドメインモデルに依存するか」(#103) など頻出。Controller が触る `OrderPlan` は**デコーダが境界で生成した型**で、ドメインロジックそのものではない。型が同じでもレイヤーの責務は分かれている。むしろ型を分けないことで、詰め替え時の情報欠落リスクが減る。
