## 本書について

本書は、Java Web アプリケーションで広く使われている「古典ドメインモデルパターン」——`@Entity` がドメインモデルを兼ね、Bean Validation がバリデーションを担い、レイヤー間を専用 DTO で詰め替えるという構成——で生じる「モデル増殖」「詰め替え過剰」「ドメインモデルの肥大化」という問題を診断し、解消するための設計書です。題材はミールス宅配サービスという架空のドメインで、実装のサンプルコードを通して段階的に考え方を示していきます。

この構成はドメイン駆動設計（DDD）そのものではなく、DDD の戦術パターンを Spring Boot で実装する際に広まった慣習の集積です。本書が問い直すのはその慣習であり、「ユビキタス言語でドメインをモデリングする」「業務の概念を型で表現する」というDDDの本質は、むしろ本書の土台にあります。「バリデーションの知識」「永続化の知識」「型変換の知識」がそれぞれあるべき場所に置かれたとき、ドメインモデルはただの `record` でよくなる——その到達点に向かう道筋を示します。

## 対象読者

本書は次のような方を想定しています。

- 古典ドメインモデルパターンを採用した Spring Boot プロジェクトで、「Form」「Command」「Entity」が増え続けて詰め替えコードに疲弊している方
- ドメインモデルに `@Entity`・`@NotNull`・状態チェックの `if` が積み重なり、責務の整理に悩んでいる方
- `record` と `sealed interface` を使った設計を実務でどう活かすか探している方
- 「ビジネスロジックはドメイン層へ」という原則を、実装の判断基準まで落とし込みたい方

反対に、DDD の戦略的設計（ユビキタス言語・境界づけられたコンテキスト）や、マイクロサービス間の統合そのものを扱う本ではありません。本書の焦点は**一つのサービス内部のレイヤー設計**であり、DDD の入門書でもありません。

## 前提知識

- Java の基礎（`record`・`sealed interface` は本文で必要な範囲を説明しますが、Java 17 以降の構文に触れた経験があるとスムーズに読める）
- Spring Boot の基礎（`@RestController`、`@RequestBody`、依存性注入などの基本用語）
- Bean Validation（`@NotNull`、`@Valid` などを見たことがある程度で十分です）
- JPA または jOOQ の名前を知っている程度

関数型プログラミングや圏論の知識は不要です。本書で触れる「合成」はコード例で示す範囲にとどめます。

## 本書の構成

本書は13章と付録で構成されています。

- **「問題の正体」（1〜2章）**: 古典ドメインモデルパターンがどう見えるか、そこに潜む問題は何かを整理します。
- **「Parse, Don't Validate」（3〜5章）**: Bean Validation の限界と、パースしてバリデーションと型変換を同時に行う Raoh デコーダのアプローチを示します。
- **「Always-Valid Layer」（6〜8章）**: 「常に正しいデータだけが流れる層」という切り口から、状態遷移の型表現と、関心を分離したドメインモデル（record）を示します。
- **「結合のバランス」（9〜11章）**: レイヤー間の詰め替え戦略と結合強度・距離の軸を整理し、デコーダがレイヤー間の接点として機能することを示します。
- **「実践」（12〜13章）**: 既存コードへの段階的な導入手順と、どの設計を選ぶかの判断基準を示します。
- **付録**: Raoh API リファレンスと参考文献。

初めて読む方は1章から順に読み進めることをお勧めします。既に Parse, Don't Validate や Always-Valid Layer に馴染みがある方は、9章から読み始めても意味が通るように書いています。

## サンプルコードについて

本書の Java サンプルは本文中に掲載しているコード断片を実際に動作するプロジェクトから抜粋・短縮したものです。環境は Java 25、Spring Boot 4、Raoh 0.5.0 です。

> **バージョンについて**: Raoh 0.5.0 は Java 25 以上を必要とします。`sealed interface` と `record` 自体は Java 17 から正式機能ですが、Raoh のビルドおよびランタイム要件として Java 25 が必要です。Spring Boot 4 系は Java 17 以上をサポートしていますが、本書のサンプルを動かすには Java 25 以上の環境をご用意ください。

なお Lombok（`@Data` など）は、古典的構成を示す Bean Validation 側のコード例でのみ使用し、ドメインモデル（record）では使いません。record 自体が不変性とアクセッサを提供するため、Lombok の出番がないからです。
