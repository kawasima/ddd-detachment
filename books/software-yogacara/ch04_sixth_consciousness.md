## 4.1　意識の働き

第六意識（mano-vijñāna）は、八識の中でわれわれが最も親しんでいる識です。日常語で「意識」と呼んでいるものに最も近く、人間の知的活動の大部分を担っています。

第2章で触れたように、第六意識には二つの働き方があります。一つは前五識と同時に働く**五倶意識**です。眼識が赤い色を知覚するのとほぼ同時に、第六意識が「リンゴだ」と概念的に判断する——この働きです。もう一つは前五識から独立して働く**独頭意識**で、想像、思考、記憶の想起、夢などがこれにあたります。

ソフトウェア設計に携わるエンジニアの知的作業は、この両方にまたがっています。業務の現場で見聞きしたことを概念に変換する作業（五倶意識的）と、頭の中で概念を操作し、モデルを構想する作業（独頭意識的）。とりわけモデリングの作業——業務から離れた場所でホワイトボードに向かい、エンティティの関係を描く——は、独頭意識の活動そのものです。

この章では、第六意識の核心的な働きである**分別**と**命名**が、ソフトウェア設計においてどのように作用するかを見ていきます。

## 4.2　分別 — モデリングという行為の本質

### データと情報の違い

第1章で、データそのものに固有の意味は内在せず、意味は解釈する側にある、という点を論じました。この構造を唯識の用語でもう少し精密に言い直してみましょう。

データは「所縁」（しょえん、ālambana）——識が認識する対象——に相当します。`"2026-04-22"` というデータは所縁として存在しますが、それだけでは「注文日」なのか「出荷予定日」なのかは定まっていません。

情報は、第六意識が所縁を**分別**した結果です。分別（vikalpa）とは、対象を概念によって区別し、カテゴリに分け、意味を付与する第六意識の働きです。`"2026-04-22"` を「注文日」として分別するのか「出荷予定日」として分別するのかは、第六意識がどのような概念的枠組みを適用するかによって決まります。

### モデリングとは何をしているのか

この視点から見ると、ドメインモデリングとは「分別の方式を定めること」だと言えます。

エンジニアが業務を分析し、エンティティや値オブジェクトを定義するとき、彼らは生のデータ（所縁）に対してどのような概念で分別するかを決めています。「この数値は金額として扱う」「この文字列はメールアドレスとして検証する」「この日付の組は期間として束ねる」——これらはいずれも、データに対する分別の方式を定める行為です。

ここで重要なのは、分別の方式は一意ではないということです。同じ業務データに対して、異なる分別が可能です。販売部門は「注文」を「売上の単位」として分別し、物流部門は「出荷の単位」として分別します。同じ所縁に対して、異なる識が異なる分別を行う。

DDDのBounded Contextは、この構造に対する実践的な対処法です。文脈ごとに分別の方式が異なることを認め、それぞれの文脈に固有のモデルを持たせる。唯識の観点からは、Bounded Contextは「識の異なりを設計に反映する仕組み」として読み替えることができます。

## 4.3　名言（名前・言語表現） — 名前を付けるということ

### 唯識における名言

ここでいう**名言**（みょうごん）は、日常語の「名言」「格言」とは関係がありません。**名前・言語表現・概念化**を指す唯識の用語です。対応するサンスクリット語としては、文献により `nāmābhilāpa`（名の言説）、`abhilāpa`（言説）、あるいは `nāman`（名）などが挙げられ、単一の原語に一意に対応させるのは必ずしも正確ではありません。本書では漢訳の「名言」を保って記述します。以降に出てくる「名言種子」「名言薫習」も、「名前や言語表現が残す種子」「言語を使うことによる薫習」という意味で、「よい言葉」とは無関係です。読み替えるときは「名前・言語表現」と置き換えると安定します。

唯識において、名言は周辺的なテーマではありません。体系の中心に位置する概念です。世親は『唯識三十頌』の冒頭で次のように宣言しています。

> 由假說我法　有種種相轉

「我」と「法」という二つの範疇は、仮の施設（prajñapti）によって言い表されているに過ぎず、そこから種々の相が現れる、という宣言です。唯識は、われわれの経験世界を支える最も基本的な二つの区切り——認識する主体としての「我」、認識される諸事象としての「法」——を、対象の側に刻まれた区切りではなく、名言によって立てられた仮の区切りとして位置づけるところから議論を始めます。この偈は「あらゆる概念」について述べたものというより、「我」と「法」という二大範疇への仮説性の宣言ですが、そこで示された姿勢——われわれの分節は名言による仮の区切りである——は、以降の唯識の議論全体を支える出発点になっています。

唯識は、この名言の働きを複数の角度から分析します。本書で参照する関連用語を、出自を分けて整理しておきます。「名言」（nāmābhilāpa）と「仮施設」（prajñapti）はインド唯識から続く原語に基づく概念です。一方、「名言種子」「名言薫習」、そして名言を**表義名言**（言語そのもの）と**顕境名言**（言語以前の認識的な分節）に分ける整理は、東アジア（中国法相教学）で発展した教学上の整理です。本書では両者を厳密に分けずに用いますが、これらは同じレイヤーの概念ではないことを念頭に置いてください。「四尋思（しじんし）」もインド系の修行論ですが、本書ではこの概念そのものは深く扱いません。

本書が第4章で名前付けに紙幅を割くのは、唯識に強引に結びつけるためではありません。**名言は唯識がもともと根本問題として扱ってきたテーマであり、その分析がソフトウェア設計の命名実践に直接届く**からです。これら四つの概念のうち、本章の後半では「名言薫習」がユビキタス言語の実践とどう接続するかを論じます（名言種子は第8章・第11章で、仮施設は第4章の本節でここまで述べた通りです）。

ここで注意すべきは、名言以前にも認識は存在するということです。第3章で見たように、前五識は概念を介さずに対象を直接的に了別（現量）します。第六意識の分別も、言語化以前の段階がありえます。名言は、すでにある認識に言語的な形を与える行為です。

しかし、名言は単なるラベル貼りではありません。名前を与えることで、認識の内容が固定化され、共有可能になり、再利用可能になります。名前のない認識は個人的で一回的ですが、名前を得た認識は概念として流通し始めます。これは強力な道具であると同時に、リスクでもあります。名前が付いた途端、われわれはその名前が指す「もの」が確固たる実体として存在するかのように思い込みやすくなるからです（遍計所執性について第7章で詳述します）。

### 命名のプロセス — 段階的な分別の深化

ソフトウェア設計における命名は、一回の行為ではなく、段階的に深化するプロセスです。Arlo Belshee は[「命名はプロセスである」](https://arlobelshee.com/good-naming-is-a-process-not-a-single-step/)と論じ、名前なし（Missing）から始まって無意味（Nonsense）、実直（Honest）、意図の表現（Intent）、ドメイン抽象（Domain Abstraction）へと深化する7段階として整理しました。ここではそのすべてに唯識の概念を当てはめるのではなく、大きな流れとして捉えます。

**初期段階（名前なし〜無意味な名前）。** 変数 `a`、`x`、`tmp`、あるいは `foo`、`bar` のような名前。コードは存在しますが、意味のある名言が与えられていない状態です。対象に対する分別がまだ粗い、あるいは分別の結果がまだ言語化されていない段階です。

**中間段階（実直な名前）。** `userList`、`isValid`、`calculateTotal` のような名前。コードが実際にやっていることをそのまま表す名前です。分別が始まり、対象の性質を言語的に捉えようとしています。しかし、この段階の名前は実装の詳細を映しており、業務の概念とは結びついていません。

**成熟段階（ドメイン抽象）。** `Order`、`ShippingAddress`、`isEligibleForHolidayDiscount` のような名前。業務ドメインの言葉で語られる名前です。分別が業務の概念に到達し、コードの世界と業務の世界が同じ言葉で結ばれます。

この段階的な深化は、第六意識による分別が洗練されていく過程です。最初は粗い分別が、対象の理解が深まるにつれて精緻になり、最終的には業務で実際に使われている概念に十分近づき、関係者が共有できる分別になっていく。ここで「業務の概念に近づく」と言うとき、「現実の写し」として完全一致を目指しているわけではありません。第12章で改めて触れるように、モデルは現実の写像ではなく、文脈の中で有効な分別の方式です。ただし、モデルが表象であるということは、業務側の客観的な制約——会計の整合性、在庫数の正確性、法令やAPI契約の遵守——を軽んじてよいということではありません。制約を満たした上で、なおそれが唯一の分別ではないという意味です。命名が設計行為であるのは、名前を選ぶことが、こうした制約のなかで分別の方式を定めることだからです。

### 名前重要 — コードで見る名言の効果

具体的なコードで名言の効果を見てみましょう。

**プリミティブの値に名前を付ける。** `int` や `String` のまま使い続けるのは、分別が粗いまま放置されている状態です。

```java
// 分別が粗い
int price = 1000;
int quantity = 5;
int total = price * quantity;

// 名言を与える
Money price = Money.of(1000, JPY);
Quantity quantity = Quantity.of(5);
Money total = price.multiply(quantity);
```

`Money` という名言を付与した途端、この値は「ただの整数」ではなくなります。通貨を持ち、負の値を許さず、他の `Money` とのみ加算できる——という分別の構造が型として表現されます。

**条件式に名前を付ける。** 複雑な条件分岐も、名言を持たない分別です。

```java
// 名言のない分別
if (order.getDate().isAfter(holiday.getStart())
    && order.getDate().isBefore(holiday.getEnd())
    && order.getTotal() > 10000) {
    applyDiscount();
}

// 名言を与える
if (order.isEligibleForHolidayDiscount()) {
    applyDiscount();
}
```

条件式にはすでにロジック（分別）がありますが、名前がないと「何を分別しているか」が読み取りにくい。名言を与えることで、分別の意図が明示され、その概念が独立したものとして共有可能になります。

## 4.4　名言薫習 — 言語が認識を形成する

### ユビキタス言語の本質

DDDの「ユビキタス言語」は、単なる用語集（glossary）ではありません。それが文字通り「言語」と呼ばれるのはなぜでしょうか。

記号論の枠組みでは、言語を四つの層に分けて分析します。

- **語彙（lexicon）** — 単語の集合。「注文」「顧客」「出荷」
- **統語（syntax）** — 語の組み合わせ規則。「顧客が注文を作成する」
- **意味（semantics）** — 語と文の意味。「注文の合計は明細の合計と一致する」（不変条件）
- **語用（pragmatics）** — 文脈による意味の変化。「注文」が販売文脈と物流文脈で異なる意味を持つ

用語集はlexiconしか持ちません。しかしユビキタス言語は、統語（データの構成規則）、意味（不変条件や状態遷移）、語用（Bounded Contextによる文脈の切り分け）のすべてを含みます。ユビキタス言語の設計とは、ドメインモデリングそのものなのです。

### 名言薫習 — 言葉が種子になる

唯識には「名言薫習」（みょうごんくんじゅう）という概念があります。梵語としては `abhilāpa-vāsanā` や `nāmābhilāpa-vāsanā` などの形で諸文献に現れます。

薫習とは、認識活動がその瞬間に阿頼耶識（第6章参照）に種子を残すことです。一回の行為でも薫習は起きます。反復によって種子が強化されるのは事実ですが、薫習の仕組みそのものは反復に依存しません。

名言薫習には二つの側面があります。一つは**表義名言薫習**——「注文」「顧客」といった言語的表現を使うことで、その言語的な分別の仕方が種子として残ること。もう一つは**顕境名言薫習**——言語を介さない認識活動であっても、対象を認識する行為そのものが種子を残すこと。ソフトウェア設計の文脈でとりわけ重要なのは前者、表義名言薫習の側面です。

ユビキタス言語の実践は、この表義名言薫習の観点から理解できます。チームが「注文」という言葉を使うたびに、各メンバーの阿頼耶識に「注文」という分別の種子が蓄積されます。同じ言語環境に繰り返し接することで、各メンバーの阿頼耶識にはそれぞれ類似の名言種子が蓄積されていくと考えられます。その結果、チーム全員が「注文」を同じように分別するようになる——というのがここでの見立てです。

ここで注意していただきたいのは、唯識はあくまで個人の識を論じる思想であり、チーム全体の認識を直接扱う枠組みではない、という点です。「同じ言語環境に置かれた複数の個人の阿頼耶識に類似の種子が並行して蓄積される」という描写は、唯識の枠組みそのものから素直に演繹できるというよりは、個人の薫習の理論に**共相種子**という別の概念を媒介させて組み立てる読み替えです。そのため、本章ではこの見立てを概略として提示するに留め、**どのような規則でこの読み替えが成立するかの構造は第8章で詳しく扱います**。それまでは「個人の薫習が媒介概念を経てチーム規模に届くという構図がある」という程度に受け取ってください。

ここからは本書独自の読み替えになりますが、ソフトウェア開発においてはさらに興味深い循環が生じます。言葉が設計者の認識を形成し、認識が設計を方向づけ、設計がコードになり、コードの中の言葉が再び設計者の認識に薫習する。唯識における種子と現行の循環（種子が現行を生み、現行が種子を薫習する）は一瞬ごとの因果ですが、ソフトウェア開発ではこの循環が言語→設計→コード→言語という多段階のプロセスとして展開されるのです。

逆に言えば、不適切な言葉は不適切な種子を植えます。「ユーザー」という曖昧な言葉を使い続ければ、「買い手」と「売り手」と「管理者」の区別が曖昧なまま設計が進みます。この循環を断ち切るには、言葉を変えること——名言を改めること——が出発点になります。

## 4.5　分別の限界 — 見えない立場

第六意識は強力な道具ですが、構造的な限界があります。

### 分別は必ず切り捨てる

分別とは、対象を概念によって区切ることです。区切るということは、区切られた外側を捨てるということでもあります。

「顧客」という概念を形成する過程で、その人間の持つ無数の側面——家族構成、趣味、感情——は捨象されます。「注文」という概念を形成する過程で、その取引の背後にある人間関係や交渉の経緯は見えなくなります。モデルとは、分別によって切り取られた断片に過ぎません。

これは避けられないことですし、ソフトウェアエンジニアの多くは「モデルは現実の簡略化である」と知識としては理解しています。しかし唯識は、分別の限界にはもう一つ、より根本的な層があると指摘します。

### 立場の不可視性

あるエンジニアが注文管理システムのドメインモデルを設計するとしましょう。データベース設計の経験が長いエンジニアには、エンティティとリレーションシップが自然に見えます。「注文」は「注文テーブル」であり、「顧客」とは外部キーで結ばれ、「注文明細」とは一対多の関係を持つ。

同じドメインを、関数型プログラミングの経験が長いエンジニアが見ると、異なるものが見えます。「注文」はイベントの列であり、「注文作成」「明細追加」「承認」という変換のパイプラインとして構造化される。

どちらのエンジニアも、自分の見方を「自分の」見方だとは感じていません。ドメインが「そうなっている」と感じています。データベースに慣れた人は、エンティティとリレーションシップが客観的に「そこにある」と感じる。関数型に慣れた人は、イベントと変換が客観的に「そこにある」と感じる。

ここに末那識の効果があります。第2章で述べたように、末那識は「自己」を掴み続ける識です。その働きの一つが、**自分の認知的立場を透明にしてしまうこと**です。末那識の我見（「これが自己だ」という見なし）は、自分の視点を「視点のない視点」——つまり客観そのもの——に変えてしまいます。

分別は常に特定の立場から行われます。しかし末那識の働きによって、その立場が分別する本人からは見えなくなります。「モデルは現実の簡略化である」と知的には理解していても、自分の分別が特定の立場から行われているということは、なかなか実感できません。自分のモデルは「簡略化した結果」に見えるのではなく、「ドメインの構造」に見えるのです。

### なぜこれが深刻なのか

これは「自分の作品への愛着」のような心理的な問題とは異なります。もっと構造的な問題です。

愛着であれば、自覚すれば対処できます。「自分はこの設計に愛着を持っている。だから冷静に判断しよう」と。しかし立場の不可視性は、自覚の対象にすること自体が困難です。なぜなら、自覚しようとする行為そのものが同じ立場から行われるからです。データベース思考で育ったエンジニアが「自分はデータベース思考に偏っているかもしれない」と反省するとき、その反省自体がデータベース思考の枠内で行われます。

DDDのBounded Contextが単なる整理の技法を超えた意義を持つのは、この立場の不可視性に対する実践的な対処だからです。「販売チームの認識」と「物流チームの認識」を別のBounded Contextに分けるということは、**一つの立場からはドメイン全体を見通せないことを、設計の構造として認めること**です。

しかし、Bounded Contextもまたある立場から引かれた境界です。この境界の引き方自体を問い直すには、何が必要なのか。第六意識の分別能力を高めるだけでは足りません。末那識が立場を不可視にする構造そのものを見る必要があります。

## 4.6　設計レビューで使う問い

本章の内容を、設計レビューで使える問いに整理しておきます。

**分別の問い：** 概念の切り出し方は、解こうとしている問題に対して妥当か？

- 「このエンティティ／集約は、業務のどの単位を切り出したものか説明できるか？」
- 「実装の都合で切り分けた境界が、業務の境界として読まれていないか？」
- 「捨てられた側面（モデルが扱っていない情報）は、どこかで取り戻せるようにしてあるか？」

**名言の問い：** この名前は、業務の言葉として通用するか？

- 「この名前は実装由来か、業務由来か？」
- 「ドメインエキスパートが、この名前を聞いて同じものを思い浮かべるか？」
- 「型・関数・変数の名前が、コード内とドキュメント・会話で一致しているか？」
- 「`User`、`Item`、`Manager` のような曖昧な総称が、本来区別すべきものを覆い隠していないか？」

**Bounded Context の問い：** 文脈の境界は、認識の異なりに即しているか？

- 「同じ言葉が、文脈ごとに別のものを指していないか（指していたら、それは別のモデルとして扱われているか）？」
- 「この境界は、誰の視点から引かれているか？」

次の第5章では、この末那識の働きがソフトウェア設計にどのような影響を及ぼすかを、正面から取り上げます。
